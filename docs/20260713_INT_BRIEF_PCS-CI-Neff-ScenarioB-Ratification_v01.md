## §1 — The problem

Algorithm v21 §7.3/§13 mandate a canonical NumPy reference whose outputs the production engine must match, pinned in `tests/golden/`. Building that reference exposed two defects:

1. **§7.2 under-determines n_eff.** The formula `n_eff = Σ(check.weight) × scaleFactor` collides with §2's locked identity inventories, whose weights sum to 1.0 by construction — making identity n_eff = scaleFactor regardless of evidence volume. Custody and Risk checks carry no defined weights at all.
2. **§12's printed CI values were not generated by §7's procedure.** Under either defensible reading of n_eff, the reference cannot reproduce them; the implied n_eff needed to match each case's CI width varies case-by-case (≈12 to ≈49) with no consistent rule. The values were hand-authored illustrations.

## §2 — The two readings

- **Scenario A (spec-literal):** quadrant weights sum to 1.0 → n_eff = scaleFactor. CIs are wide everywhere and **three §12 tiers flip**: §12.1 Gold→Silver, §12.3 Gold→Silver, §12.4 Silver→Bronze. The prior TypeScript engine was mathematically Scenario A and preserved the doc tiers only through an unspec'd recentre-on-composite step.
- **Scenario B (count-based):** every check, custody event, and forensic test that ran counts one trial → n_eff = (#checks) × scaleFactor. **All seven §12 tiers hold from honest raw quantiles**, with no recentring.

## §3 — Decision (ratified)

**Scenario B is binding.** n_eff counts evidence: identity = attributes assessed; custody = documented custody events; material = forensic checks run (fractional missing-weight still drives §7.2 variance inflation); risk = registry sweeps run (1 at v01).

**Rationale — evidence accumulation, not tier preservation.** More documents ⇒ more trials ⇒ tighter posterior ⇒ narrower CI. That is the epistemically correct behaviour for "intelligence, not insurance": confidence must be earned by evidence volume, and a thin file must read wide. Scenario A makes the CI width a function of the domain alone — a two-document file and a forty-document file would carry identical uncertainty, which is wrong. That all seven §12 tiers survive under B is a consequence, not the justification.

The **recentre-on-composite step is removed** from the engine. §7.3's raw Monte-Carlo quantiles with `point = mean(samples)` stand as written. The displayed composite remains the algebraic weighted sum.

## §4 — The reproducibility contract (ruling of 13 July 2026)

Bit-identical cross-language streams — §7.3's original ambition — are **unattainable wherever libm enters the draw path**, and NumPy itself is not bit-stable across platforms:

- NumPy's beta sampler calls the **system** `log`/`pow` inside its rejection loops. Apple's arm64 libm is not correctly rounded (measured: 2 of 538 sampled log arguments differ from the true rounding); V8/Node ships its own fdlibm port; glibc differs again. ~1-ulp sample drift between any two platforms is structural.
- Separately, NumPy's macos-arm64 wheel compiler-fuses `1.0 + c·X` in the gamma sampler (fmadd); baseline x86-64 wheels cannot.

**Contract:** CI bounds (point, lo, hi) are reported, compared, and SHA-256-hashed at **2 decimal places, round-half-even on the exact value** (CPython `f"{x:.2f}"` semantics; the TS engine implements the identical rounding exactly via integer arithmetic). **The tier maps on the rounded lower bound.** Sub-2dp drift is out of contract scope by design; a 1-ulp difference at ~1e-13 cannot move a 2-dp digit except at an exact decimal tie, which is detectable and has probability ≈ 0. The pinned golden (`pcs-golden-v21.json`) contains only contract-precision values plus platform-independent inputs (posterior parameters, n_eff, seeds), so it regenerates identically on any platform.

Everything deterministic-by-construction IS bit-exact and test-locked: seed derivation (SHA-256 → LE uint64), SeedSequence spreading, PCG64 stream, ziggurat normal/exponential layers, gamma/beta draw paths, pairwise-summation mean — verified layer-by-layer against NumPy 2.0.2 in `packages/pcs-core/np-parity.test.ts` (vectors generated on macos-arm64; the vectors file is platform-tagged diagnostic material, not the contract).
